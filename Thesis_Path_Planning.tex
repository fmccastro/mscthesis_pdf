\section{Traversability map computation}
\label{section:costMap}

As it is mentioned in subsection \ref{subsection:worldModelling}, there is no explicit guideline on how to generate a traversability map. In this thesis, several approaches are compared. First, the state-of-the art algorithm from \citet{Amorim2014} is reviewed and evaluated. Then, this algorithm is reformulated to take into account other metrics such as terrain inclination and roughness \cite{Ishigami2007PathPlanning}. Finally, these two maps are merged to yield a conservative, worst-case version of them, followed by its refinement, as it is shown in figure \ref{fig:travmapsgenerationflowchart}.

\subsection{Height minimization}
\label{subsection:heightMinimization}

A similar constrained optimization problem to the one from \cite{Amorim2014}, which minimizes the height of the robot at each cell is described here. In this case, a wheeled vehicle is used instead of a tracked one. The yaw angle is set as a fixed variable and the initial guess $\boldsymbol{w}_0$ of the problem at every cell is set by computing an \ti{a-priori} approximation to the optimal solution.

\subsubsection*{Initial guess}

Provided that the yaw angle $\yaw$ and the $\x$ and $\y$ position coordinates are known, each wheel point is projected to $\Map$. Then, from the group of retrieved contact points, the lowest height one is discarded, and a normalized, ortogonal vector to a plane which contains these points, is computed. Finally, the roll and pitch of this plane are given by \refeq{eq:rollPitchInitialGuess}, where $\mathbf{\hat{n}}$ denotes the normalized ortogonal vector to the above mentioned plane. In case any of these angles are higher than $\roll_{max}$ or $\pitch_{max}$, the respective angle is set to the highest admissable value, so that the initial guess is feasible. The elevation initial guess $\z_0$ is set as the sum of the projection of the robot center of mass point to $\Map$ plus the robot COM geometric height, to avoid any possible intrusion of the vehicle hull into the ground.

\begin{equation}
	\begin{aligned}
		\roll &= \arcsin \left( \sin(\yaw) \, \hat{n}_x -\cos(\yaw)\, \hat{n}_y \right) \\
		\pitch &= \arccos \left( \frac{\hat{n}_z}{\cos\left( \roll \right)} \right) \\
	\end{aligned}	
	\label{eq:rollPitchInitialGuess}
\end{equation}

\subsubsection*{Nonlinear programming}

The minimization of the vehicle's center of mass is expressed as a nonlinear optimization problem (NLP) in \refeq{eq:minHeight}, where $\boldsymbol{\Orientation}_{max}$ stands as the limits for each euler angle. $\mathbf{p}_{3,\,k}$ stands as the $k^{th}$ hull point position, which is defined with respect to $\WorldFrame$, by taking into account the robot geometry. $[\x, \y, \yaw]^T$ are fixed variables whereas $\left[ {\z},\, {\roll},\, {\pitch}\right]^T$ are set as optimization variables. Each hull point included in \refeq{eq:minHeight} adds an additional constraint by stating each hull point must be above or at contact with the terrain. In case less than three hull points (each one from a different wheel) are at contact with the terrain, the terrain inclination returned by algorithm \nameref{alg:slopeMinHeight} is $\frac{\pi}{2}$. This expresses the fact that the robot will stand at an unstable position, so this position is regarded as an undesirable traversable position.  On the other hand, the terrain inclination is given by \refeq{eq:inclination}, which returns the angle between a normalized vector with direction $\hat{\BodyFrame}_z$ and the normalized vector $\hat{\WorldFrame}_z$ \cite{Amorim2014}. Although algorithm \nameref{alg:slopeMinHeight} can be extended to any kind of wheeled-mobile robot configuration, only four wheel mobile robots are considered in this thesis, as it is mentioned in section \ref{section:problemFormulation}.

	\Else

	\EndIf

	\EndFor
	\State \Return $\FunctionTraversability_d(\cdot)$ 
	\EndFunction

Then, algorithm \nameref{alg:slopeMinHeightMap} extends the application of algorithm \nameref{alg:slopeMinHeight} to every cell of the elevation grid. Moreover, a convex area $\Robot_{\mathbf{p}_2}$ (\refeq{eq:robotHullArea}) centered at $\mathbf{p}_2$ (typically the robot's COM), must capture the robot's geometric hull $\mathcal{H}_2$ projection on $\Map$ and lie inside the map domain. If this condition is not met, the terrain inclination angle assigned to $\FunctionTraversability_d(\cdot)$ is $\frac{\pi}{2}$. Therefore, positions at which the assigned terrain inclination is $\frac{\pi}{2}$ are considered to be untraversable. To sum up, the higher the inclination, the less desirable it is to pass through the respective position.

The discrete elevation map $\FunctionElevation_d(\cdot)$ shall have such dimensions that the cell divison length is smaller than the order of magnitude of the wheel diameter.

\subsubsection*{Variable yaw}

The terrain traversability may also be assessed by finding the maximum inclination at each position by saving the maximum retrieved inclination among several yaws. Algorithm \nameref{alg:slopeMinHeightMap} lacks a thorough evaluation of the terrain irregularity, as this feature changes depending on the mobile robot heading. In order to tackle this absence of completeness, algorithm \nameref{alg:slopeMinHeightMap} is reformulated into algorithm \nameref{alg:slopeMinHeightMapVarYaw} to address this issue.

\begin{algorithm}[H]
	\caption{\footnotesize \textsc{MAPSLOPE}}\label{alg:slopeMinHeightMapVarYaw}
	\centering
	\footnotesize
	\begin{algorithmic}[1]
	\Function{inclinationvaryaw}{$\boldsymbol{\Orientation}_{max}, \, \FunctionElevation(\cdot), \,n_x,\, n_y$}
	\ForEach{$\{i,\,j\} \in \{0,\, \ldots, \, n_x - 1\} \times \{0,\, \ldots, \, n_y - 1\}$}
	\ForEach{$\yaw_k \in \mathcal{Y}$} \Comment{change heading}
	\State $\boldsymbol{w}_0 \gets \textsc{initialguess}(x_i, \, y_j, \, \yaw_k)$ \Comment{feasible initialization}
	\If{$\Robot_{\mathbf{p}_2} \subset \Map_2$}
	\State $\TraversabilityCost_k,\, \left( \boldsymbol{w}^* \right)_k \gets \textsc{terrainslope}(\boldsymbol{\Orientation}_{max}, \, \FunctionElevation(\cdot), \, \boldsymbol{w}_0, \, x_i, \, y_j, \, \yaw_k)$ \Comment{algorithm \nameref{alg:slopeMinHeight}}
	\Else
	\State $\TraversabilityCost_k \gets \frac{\pi}{2}$
	\EndIf
	\EndFor
	\State $\TraversabilityCost \gets \text{max}\left( \TraversabilityCost_k \right)$ \Comment{save maximum inclination}
	\State $\FunctionTraversability_d(i,\, j) \gets \TraversabilityCost$ \Comment{assign $c$ to respective cell in discrete traversability map}
	\EndFor
	\State \Return $\FunctionTraversability_d(\cdot)$
	\EndFunction
	\end{algorithmic}
\end{algorithm}

In algorithm \nameref{alg:slopeMinHeightMapVarYaw}, $\mathcal{Y} = \lbrace -\pi,\, \ldots,\, -\pi + k \alpha, \, \ldots, \, \pi - \alpha \rbrace$, where $\alpha$ stands as the step angle between consecutive elements of $\mathcal{Y}$.

\subsection{Maximum terrain roughness}
\label{subsection:maxTerrainRoughness}

Terrain traversability may also be evaluated by inspecting terrain roughness around each position of the map, where the goal is to evaluate the maximum terrain uneveness at each position of the map. In this thesis, terrain roughness at each position of $\Map$ is defined as the standard deviation of elevations inside $\RobotHullArea_2$.

\begin{equation}
	\bar{\FunctionElevation}_d(i, j) = \frac{\sum_{\x_i,\,\y_j \in \RobotHullArea_2} \FunctionElevation(x_i, \, y_j)}{n(\underline{\RobotHullArea}_2)},\; \underline{\RobotHullArea}_2 :=  \forall \left( \x_i,\, \y_j \right) \in \RobotHullArea_2
	\label{eq:averageHeight}
\end{equation}

\noindent, where $\underline{\RobotHullArea}_2$ denotes a finite set of points from $\FunctionElevation_d(\cdot)$ inside $\RobotHullArea_2$ and $\bar{\FunctionElevation}_d(i, j)$ stands as the average of elevations of points inside $\RobotHullArea_2$. Then, terrain roughness is defined as,

\begin{equation}
	\FunctionTraversability_d(i,\,j) = \sqrt{\frac{\sum_{\x_i,\,\y_j \in \RobotHullArea_2} \left( \FunctionElevation_d(i,\,j) - \bar{\FunctionElevation}_d(i,\,j) \right)^2 }{n(\underline{\RobotHullArea}_2)}}
	\label{eq:terrainRoughness}
\end{equation}

The set of points inside $\underline{\RobotHullArea}_2$ depends on the yaw angle of the mobile robot, so $\FunctionTraversability_d(i,\,j)$ is computed with respect to a given heading. Similarly to what was done in algorithm \nameref{alg:slopeMinHeightMapVarYaw}, the terrain roughness assigned to the respective cell is the maximum roughness among the set of yaws $\mathcal{Y}$. Finally, a maximum terrain roughness may be computed by implementing algorithm \nameref{alg:maxRoughnessMap}.

\begin{algorithm}[H]
	\caption{\footnotesize \textsc{MAPROUGHNESS}}\label{alg:maxRoughnessMap}
	\centering
	\footnotesize
	\begin{algorithmic}[1]
	\Function{roughness}{$\boldsymbol{\Orientation}_{max}, \, \FunctionElevation(\cdot), \,n_x,\, n_y$}
	\ForEach{$\{i,\,j\} \in \{0,\, \ldots, \, n_x - 1\} \times \{0,\, \ldots, \, n_y - 1\}$}
	\If{$\Robot_{\mathbf{p}_2} \subset \Map_2$}
	\ForEach{$\yaw_k \in \mathcal{Y}$} \Comment{change heading}
	\State $\TraversabilityCost_k \gets \text{equation \eqref{eq:terrainRoughness}}$
	\EndFor
	\Else
	\State $\TraversabilityCost_k \gets \texttt{None}$
	\EndIf
	\State $\TraversabilityCost \gets \text{max}\left( \TraversabilityCost_k \right)$ \Comment{save maximum roughness}
	\State $\FunctionTraversability_d(i,\, j) \gets \TraversabilityCost$
	\EndFor
	\State \text{Replace cells assigned with $\texttt{None}$ by max} $\left( \FunctionTraversability_d(\cdot) \right)$
	\State \Return $\FunctionTraversability_d(\cdot)$ 
	\EndFunction
	\end{algorithmic}
\end{algorithm}

\begin{remark}
In case $\RobotCircleHullArea_2 \not\subset \Map_2$, the respective cell is assigned with $\texttt{None}$. At the end, these assignments are replaced by the maximum roughness among all cells, in order to enhance the fact that these cells are meant to be among the most untraversable ones.
\end{remark}

\subsection{Worst-case scenario}
\label{subsection:worstCaseScenario}

A more conservative traversability map can be created by assigning to every cell, the highest value among the traversability maps computed by algorithms \nameref{alg:slopeMinHeightMapVarYaw} and \nameref{alg:maxRoughnessMap} at the respective cell. This approach provides a \tb{joint awareness} of terrain inclination and roughness on a single traversability map. In order to accomplish that, the traversability maps that come into play are normalized in order to enable a fair comparison among maps.

\subsubsection*{Traversability map refinement}
\label{subsubsec:costMapRefinement}

In order to classify high cost borderline areas as undesirable, a refinement of the worst-case scenario map is made, so that a mobile robot avoids a close traverse to undesirable positions. Algorithm \nameref{alg:refinedMap} presents a way to code this refinement method, by increasing the cost of areas around positions with a cost higher than a defined threshold, $c_{max}$. At each point with cost higher than $c_{max}$, a \tb{decreasing, isometrical source} radiates up to a distance $r_s$ from the respective point. To every point that is inside the source, the existing cost is replaced by the source cost $\TraversabilityCost_s \left(\cdot \right)$ in case this one is higher.

\begin{equation}
\TraversabilityCost_s \left(x \right) = \begin{cases} -\left( \frac{x}{r_s} \right)^2 + 1 \, , \hspace{5mm} 0 \leq x \leq r_s\\
0,\,x > r_s \end{cases}
\label{eq:sourceCost}
\end{equation}

\begin{algorithm}[H]
	\caption{\footnotesize \textsc{REFINEWORSTCASE}}
	\label{alg:refinedMap}
	\centering
	\footnotesize
	\begin{algorithmic}[1]
	\Require $\boldsymbol{\Orientation}_{max}, \, \FunctionElevation(\cdot), \,n_x,\, n_y, \, r_s, \, C$
	\State $\FunctionTraversability_{d, \, inc}(\cdot) \gets \textsc{inclinationvaryaw}(\boldsymbol{\Orientation}_{max}, \, \FunctionElevation(\cdot), \,n_x,\, n_y)$ \Comment{run algorithm \nameref{alg:slopeMinHeightMapVarYaw}} 
	\State $\FunctionTraversability_{d,\,r}(\cdot) \gets \textsc{roughness}(\boldsymbol{\Orientation}_{max}, \, \FunctionElevation(\cdot), \,n_x,\, n_y)$ \Comment{run algorithm \nameref{alg:maxRoughnessMap}}
	\State $\FunctionTraversability_{d, \, inc}(\cdot),\, \FunctionTraversability_{d,\,r}(\cdot) \gets \frac{\FunctionTraversability_{d, \, inc}(\cdot)}{\text{max}\left( \FunctionTraversability_{d, \, inc}(\cdot) \right)}, \, \frac{\FunctionTraversability_{d,\,r}(\cdot)}{\text{max} \left( \FunctionTraversability_{d,\,r}(\cdot) \right)}$ \Comment{normalize maps}
	\ForEach{$\{i,\,j\} \in \{0,\, \ldots, \, n_x - 1\} \times \{0,\, \ldots, \, n_y - 1\}$}
	\State $\FunctionTraversability_d(i,\, j) \gets \text{max} \left(\FunctionTraversability_{d, \, inc}(i,\,j), \hspace{1mm} \FunctionTraversability_{d,\,r}(i,\,j) \right)$ \Comment{generate worst-case map}
	\EndFor
	\ForEach{$\{i,\,j\} \in \{0,\, \ldots, \, n_x - 1\} \times \{0,\, \ldots, \, n_y - 1\}$}
	\If{$\FunctionTraversability_d(i,\, j) \geq C$}
	\State \text{Add $\left( i, \, j \right)$ to $p$} \Comment{add point with cost above or equal to $C$ to list}
	\EndIf
	\EndFor
	\ForEach{$\{i,\,j\} \in p$}
	\ForEach{$\{x_{i_2},\, y_{j_2}\} \in \{x_i - r_s,\,\ldots,x_i + r_s\} \times \{y_j - r_s,\,\ldots, y_j + r_s\}$}
	\If{$\{x_{i_2},\,y_{j_2}\} \in \Map_2$}
	\State $d \gets \sqrt{\left( x_{i_2} - x_i \right)^2 + \left( y_{j_2} - y_j \right)^2}$ \Comment{compute distance between $\left(i,\,j\right)$ and $\left(i_2,\,j_2\right)$}
	\State $T_d\left(i_2,\, j_2 \right) \gets c_s(d)$ \Comment{add \refeq{eq:sourceCost} result to list of source costs which concern $\left(i_2,\, j_2 \right)$}
	\EndIf
	\EndFor
	\EndFor
	\ForEach{$\{i,\,j\} \in \{0,\, \ldots, \, n_x - 1\} \times \{0,\, \ldots, \, n_y - 1\}$}
	\State $\FunctionTraversability_d\left(i,\, j\right) \gets \text{max}\left( \FunctionTraversability_d(i,\, j),\,T_d\left(i,\, j\right) \right)$ \Comment{assign highest cost among source costs and existing cost}
	\EndFor
	\end{algorithmic}
\end{algorithm}

The origin of each elevation map is considered to be an inertial world frame $\WorldFrame$, which is placed at the center of each elevation map, as it is depicted in the picture of figure \ref{fig:elevationmapdimensions}. Typically, the original files of elevation maps come as \texttt{.tif} or \texttt{.png} with grayscale encoding. Then, a suitable rescaling shall be made in order to yield a large enough map, with a meters per pixel discretization such that the wheel diameter is of the same magnitude order. Moreover, the maximum height shall be chosen in such a way that non-negligible crossable slopes are on the map.

Each algorithm is coded in \texttt{Python}, whereas the NLP presented in \refeq{eq:minHeight} is implemented with the casADi numerical optimization tool \cite{Andersson2018}, where IPOPT \cite{wachter2006implementation} is selected as solver. Two different sets of simulations are performed for each map: the first only considers a single contact point at each wheel, whereas the second one takes a set of five hull points at each wheel (figure \ref{fig:hullPoints}).

\subsection{Discussion}
\label{subsection:costMapsDiscussion}

Algorithms \ref{alg:slopeMinHeightMap} to \ref{alg:refinedMap} were built in such a way that the output map assigns higher costs to hilly or irregular terrain. On the other hand, it was required that the output map captures the terrain height gradient, although it is also advisable to assign higher costs to positions which are close to untraversable areas. Then, a balance between a traversability map that leads to the computation of a safe path and an accurate capture of the terrain morphology comes into play.

The analysis of figure \ref{fig:costMapsCase1Map1} and tables \ref{tab:resultsMap_1_Case1_2_Alg2_3} and \ref{tab:results_Map1_Case1_2_Alg_4_5} express the fact that the generated maps capture well the terrain morphology as the correlation between the height gradient and the cost is very high on all cases. However, that correlation decreases a lot when algorithm \nameref{alg:refinedMap} is applied, due to the fact that the map refinement assigns higher costs to positions which are close to untraversable positions. This is the price to pay in case one wants to ensure the avoidance of hazardous positions. The correlation $\nabla \FunctionElevation(\cdot) \text{ vs }\FunctionTraversability(\cdot)$ will depend on on the parameters $r_s$ and $C$. In case $r_s$ is high and $C$ is low, the source that radiates from each point with cost higher or equal to $C$ is larger and applied to a larger set of points, which causes the correlation $\nabla \FunctionElevation(\cdot) \text{ vs }\FunctionTraversability(\cdot)$ to be smaller.

\section{Optimal path planning}
\label{section:optimalPathPlanning}

Up to now, it is possible to compute a potential flow with respect to any cost map generated based on the algorithms presented in section \ref{section:costMap}. Function $\speedFunction\cdot$ from equation \ref{eq:eikonalEquation} is replaced by a speed map. Then, the solution of the Eikonal equation outputs a function $\timeFunction\cdot$ that holds the time it takes to reach the goal position from any position $\mathbf{p} \in \Map_2$. Therefore, $\nabla \timeFunction\cdot$ enables the computation of an optimal path towards the selected goal. $\timeFunction\cdot$ does not possess local  minimums, so there's no risk the computation of the optimal path gets stuck.

However, $\speedFunction\cdot$ is not directly replaced by any cost map generated by algorithms \ref{alg:slopeMinHeightMap} to \ref{alg:refinedMap}, because lower speed maps must match higher costs. So, these cost maps must be flipped into a speed map beforehand.

\begin{equation}
	\speedFunction{x_i,\,y_j} = e^{\frac{\ln\left(k_1\right)}{k_2} \cdot \FunctionTraversability_d \left(x_i,\, y_j\right)}
	\label{eq:cost2SpeedConversion}
\end{equation}

Although this conversion can take many forms, it must comply with the fact that higher costs match lower speeds, so an exponential function of the type $e^{-kx}$ is chosen in this thesis. Equation \ref{eq:cost2SpeedConversion} is a particular case of $e^{-kx}$ where $-k = \frac{\ln\left(k_1\right)}{k_2}$ in order to enforce that if $\FunctionTraversability_d \left(x_i,\, y_j\right) \geq k_2$, it comes that $\speedFunction{x_i,\,y_j} \leq k_1$. The choice of $k_1$ and $k_2$ is made in such a way, so that the flipping of higher costs from $\FunctionTraversability_d \left(x_i,\, y_j\right)$ gets more penalized on the respective $\speedFunction{x_i,\,y_j}$.

Each value that is assigned to every cell on the cost map expresses the fact that it is safer or riskier to pass by that position. If $\speedFunction\cdot$ presents a high (low) value to some cell of the cost map, that means if is safer (riskier) to cross that position. Unlike occupancy maps, speed maps can take any non-negative value at each cell. On both maps, if zero is assigned to some position, that position is \ti{seen} as an obstacle. However, one may enlarge the set of obstacles on $\Map_2$ by \ti{masking} (subsection \ref{subsection:implementationfm}) on $\speedFunction{\cdot}$ the positions that $\FunctionTraversability_d \left( x_i,\,y_j \right) \geq c_{max}$, where $c_{max}$ denotes the maximum admissable cost for traversal.

\subsection{Implementation}
\label{subsection:implementationfm}

Algorithm \ref{alg:speedMapGeneration} sums up the speed map generation steps, whereas algorithm \ref{alg:optimalPathGeneration} encapsulates the optimal path generation method described in the previous section. Note that $N$ denotes the path length. Algorithm \ref{alg:speedMapGeneration} is meant to be run offline, but algorithm \ref{alg:optimalPathGeneration} is not, so the optimal path generation was coded in \texttt{C} in order to speed up its execution and allow for greater time margins on the iteration window. Then, function \textsc{getoptimalpath($\cdot$)} is called in \texttt{Python} using the well-known \texttt{ctypes} package.

The computation of a discrete solution $\speedFunction\cdot$ for the Eikonal equation is made by applying the method \texttt{travel\_time()} from the \texttt{scikit-fmm} Python package\footnote{scikit-fmm package: \texttt{\url{https://github.com/scikit-fmm/scikit-fmm}} (accessed on 23 of March 2026)}. Finally, the optimal path is obtained by repeatedly finding the minimum time position among the newest path position, until the goal position $G$ is reached or if the path surpasses its neighborhood, $\neighborhood(G)$.

As it was noted in the previous section, this method is only valid in case the map discretization is of the same magnitude order of the wheel radius. Table \ref{tab:optPathparams} presents the parameters which were chosen to generate the results presented in subsection \ref{subsection:discussionfm}.

\begin{algorithm}[H]
	\caption{\footnotesize Potential flow generation}
	\label{alg:speedMapGeneration}
	\centering
	\footnotesize
	\begin{algorithmic}[1]
	\Function{getpotentialflow}{$\FunctionTraversability_d \left( \cdot \right), \,n_x, \,n_y, \, k_1,\, k_2, \,c_{max}$}
	\ForEach{$\{i,\,j\} \in \{0,\, \ldots, \, n_x - 1\} \times \{0,\, \ldots, \, n_y - 1\}$}
		\If{$\FunctionTraversability_d \left( x_i,\,y_j \right) \geq c_{max}$}
			\State $\text{Mask} \; \speedFunction{x_i,\,y_j}$ \Comment{Label this position as an obstacle.}
		\Else
			\State $\speedFunction{x_i,\,y_j} \gets \text{equation \ref{eq:cost2SpeedConversion}}$ \Comment{Flip cost map to speed map.}
		\EndIf
	\EndFor
	\State $\timeFunction{\cdot} \gets \text{equation \ref{eq:eikonalEquation}}$ \Comment{Solve eikonal equation}
	\State \Return $\timeFunction{\cdot}$
	\EndFunction
	\end{algorithmic}
\end{algorithm}

\begin{algorithm}[H]
	\caption{\footnotesize Optimal path generation}
	\label{alg:optimalPathGeneration}
	\centering
	\footnotesize
	\begin{algorithmic}[1]
	\Function{getoptimalpath}{$\timeFunction{\cdot}, \,S, \,G, \,N$}
	\If{$S \in \ObstacleSpace \; \text{or} \; G \in \ObstacleSpace$}
		\State \Return $\text{empty path}$ \Comment{Reject path generation in case S or G positions are labeled as obstacles.}
	\EndIf
	\State $\left( x_i,\,y_j \right) \gets S,\, \text{index} \gets 0$ \Comment{Generate optimal path from starting position S}
	\While{$\left( x_i,\,y_j \right) \notin \neighborhood(G) \text{ and index} \leq N$}
		\State $\left( x_i,\,y_j \right) \gets \textsc{getposition} \left( \textsc{min} \left( \timeFunction{ \partial\left( x_i,\,y_j \right) } \right) \right)$ \Comment{Get minimum time position among contiguous cells.} 
		\State $\text{path.append}\left( x_i,\,y_j \right)$ \Comment{Append minimum time position to path.}
		\State $\text{index} \gets \text{index + 1}$
	\EndWhile
	\While{$\text{index} \leq N$}
		\State $\text{path.append}\left( x_i,\,y_j \right)$ \Comment{Fill path with last position until $N+1$ positions are gathered.}
		\State $\text{index} \gets \text{index + 1}$
	\EndWhile
	\State \Return $\text{path}$
	\EndFunction
	\end{algorithmic}
\end{algorithm}

Due to the fact that \textsc{getoptimalpath($\cdot$)} returns \texttt{None} in case $S \in \ObstacleSpace$, one should take care that $\ObstacleSpace$ is not scattered along the map, but rather is restricted to \ti{blocks} of obstacles, in order to prevent frequent rejections of an optimal path computation.
